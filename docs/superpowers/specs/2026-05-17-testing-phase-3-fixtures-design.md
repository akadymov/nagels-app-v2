# Testing strategy — Phase 3 (Fixtures + POC scenario) Design

## Context

Phases 1 (foundation) and 2 (local Supabase + isolated Expo on `:8082`) shipped on `main` 2026-05-16. The four-tier suite (`unit` / `smoke` / `scenario` / `end-to-end`) defined in `2026-05-16-testing-strategy-design.md` now has the orchestration plumbing (`globalSetup` / `globalTeardown` / `LOCAL_SUPABASE=1`) and one working spec (`tests/e2e/sp-game.spec.js`, ~22 min).

Phase 3 introduces the **scenario tier** as a proof-of-concept: one fixture helper and one spec that together prove the seeding → assertion → cleanup pattern works end-to-end. Phase 5 will mass-produce remaining scenarios using the same pattern; Phase 6 will expand `players.ts` for multi-context end-to-end specs. Phase 3 stays deliberately small.

## Problem

The `e2e` tier (a full Hard-bot game) is slow (~22 min) and brittle for verifying specific in-game UI states like "no-trump hand is dealt and shows the right badges". A spec that needs to assert mid-game UI either (a) plays through hands to reach the state — slow and flaky — or (b) seeds the state directly. The scenario tier exists for (b).

We have no fixture code today. The `sp-game.spec.js` helpers (`tryBet`, `tryPlay`, `dismissTipIfAny`) are inlined and not reusable from other specs.

## Goals

- One working `seedGame()` helper that reaches `notrump-hand-5` (hand 5 of 20, no-trump variant) by driving production code paths.
- One working spec (`notrump-deal.spec.ts`) that uses the helper, asserts the dealt state, and cleans up.
- A repeatable pattern Phase 5 can extend to other scenarios without architectural changes.
- A scenario project in Playwright that runs independently of `e2e`.

## Non-goals

- Additional scenarios (`last-hand-mid-play`, `game-finished`, `with-spectator`, `mid-hand-disconnect-ready`, `host-still-in-game`) — Phase 5.
- Multi-context Playwright helpers in `players.ts` beyond the minimum needed for one human in the seeded room — Phase 6.
- Extracting `sp-game.spec.js` helpers (`tryBet` / `tryPlay` / `dismissTipIfAny`) into `tests/fixtures/actions.ts` — Phase 6 (when the e2e tier needs reuse too).
- Smoke tier — Phase 4.
- CI integration — out of scope until Phase 8.
- Any test-only production-schema additions (no `test_run_id` column, no `fast_forward_to_hand` RPC unless Phase 3 implementation discovers there's no other way to seat bots; see "Risks").

## Architecture

```
tests/
├── fixtures/
│   ├── seed.ts              — seedGame({ scenario: 'notrump-hand-5' })
│   ├── players.ts           — openPlayerContext(browser, { session, runId })
│   ├── cleanup.ts           — cleanupRun(runId)
│   └── supabaseAdmin.ts     — lazy service-role client (reads supabase status)
├── scenario/
│   └── notrump-deal.spec.ts — POC spec
├── e2e/                     — unchanged (sp-game.spec.js)
└── playwright/              — unchanged (global-setup / global-teardown / local-backend)
```

`seedGame` drives production edge functions through the local Supabase stack — `createRoom`, `joinRoom` (or whichever path seats bots), `startGame`, then `placeBet` / `playCard` for the human seat through hands 1-4. No direct INSERTs into `rooms` / `hands` / `bets` / `tricks` / `dealt_cards` — those tables stay reachable only through the same code paths production uses, so seed correctness can never drift from production correctness.

`cleanupRun` is the only fixture that bypasses edge functions: it deletes via SQL through the service-role client, relying on the existing FK cascades. That's a deliberate trade — cleanup needs to work even when production code paths are broken.

### State seeding — `tests/fixtures/seed.ts`

```ts
export type SeedScenario = 'notrump-hand-5';

export interface SeededGame {
  runId: string;          // UUID v4, per call
  roomCode: string;       // generated by createRoom
  hostSession: {
    accessToken: string;
    refreshToken: string;
    userId: string;
  };
}

export async function seedGame(opts: {
  runId: string;
  scenario: SeedScenario;
}): Promise<SeededGame>;
```

Behaviour for `scenario: 'notrump-hand-5'`:

1. Anonymous-auth a fresh user via the local Supabase URL; set `display_name = test_<runId>_p1` in user metadata.
2. Call `game-action / createRoom` with `silent: true`, `playerCount: 4`, `difficulty: 'Hard'` (or whatever the SP path uses to indicate "fill empty seats with Hard bots"). `silent` suppresses Telegram. The function returns a `roomCode`.
3. If bot seats are not filled by `createRoom`, the implementation plan identifies the path that does it (likely `startGame` with a flag, or a separate `addBots` action). This question is settled in Step 1 of the plan, not the design — the API shape above doesn't change either way.
4. Call `game-action / startGame`.
5. Game loop until `hand_number === 5 AND phase === 'betting'`:
   - Read snapshot via the same `game-action` snapshot endpoint the client uses.
   - When it's the host's turn to bet, call `placeBet` with the first legal bet value.
   - When it's the host's turn to play, call `playCard` with the first legal card.
   - Bots play themselves server-side — the seed never invokes bot actions.
   - Poll interval: 250ms.
6. Return `{ runId, roomCode, hostSession }`.

Expected wall-clock: 30-60s for the 4-hand fast-forward. This pushes the per-scenario-spec budget from the spec's original "~5s" to ~35-65s; the scenario-tier overall budget (5-8 min) still has plenty of headroom for the eventual ~6 specs.

The first-legal-bet / first-legal-card choice is deterministic but doesn't necessarily produce interesting in-game state — that's fine for hand 5 because we only assert what's dealt, not how prior hands ended. Later scenarios that care about prior hand outcomes will need richer seed parameters (Phase 5 problem).

### Player context — `tests/fixtures/players.ts`

```ts
export interface PlayerContextHandle {
  page: Page;
  context: BrowserContext;
}

export async function openPlayerContext(
  browser: Browser,
  opts: {
    session: SeededGame['hostSession'];
    roomCode: string;
    runId: string;
  },
): Promise<PlayerContextHandle>;
```

Behaviour:

1. `browser.newContext()` with the same mobile viewport / userAgent / hasTouch as the e2e config (factored once at the top of the file).
2. Navigate to the app base URL.
3. Use `page.addInitScript` (or `context.addCookies` / `localStorage` injection) to seed the Supabase auth session before the app's React tree mounts. Implementation plan identifies which Supabase JS client storage key to write to (typically `sb-<projectRef>-auth-token`).
4. Navigate to the room. The plan settles whether the room deep-link is `/r/<code>` or a query param or "go to lobby and click join". The contract here: by the time `openPlayerContext` returns, the page is loaded in the room view as the seeded user.

Phase 6 will extend this with `players(browser, n)` for multi-context orchestration; for Phase 3 we only need one human in one context.

### Cleanup — `tests/fixtures/cleanup.ts`

```ts
export async function cleanupRun(runId: string): Promise<void>;
```

Behaviour: connects via service-role client and runs

```sql
DELETE FROM rooms
WHERE id IN (
  SELECT room_id FROM room_sessions
  WHERE display_name LIKE $1
);
```

with `$1 = 'test_<runId>_%'`. FK cascades handle `hands`, `bets`, `tricks`, `dealt_cards`, `room_players`, `room_sessions`, `room_spectators`, `game_events`. Failed tests leak rows on purpose — the prefix makes them greppable for manual debug.

The implementation plan validates that all relevant tables actually cascade from `rooms`. If any don't, `cleanupRun` deletes from those directly first. No new FK constraints are added.

### Service-role client — `tests/fixtures/supabaseAdmin.ts`

```ts
export function supabaseAdmin(): SupabaseClient;
```

Lazy singleton. Reads `supabase status -o json` once (via `child_process.execSync`), pulls `API_URL` and `SERVICE_ROLE_KEY`, constructs a client with `autoRefreshToken: false`, `persistSession: false`. Reuses `assertLocalUrl` from `tests/playwright/local-backend.ts` to refuse to run if the URL is anything other than `127.0.0.1` / `localhost`.

This must never be imported from a non-test file. The `tests/fixtures/` directory is the boundary; the impl plan adds an eslint or grep guard if cheap.

### Scenario spec — `tests/scenario/notrump-deal.spec.ts`

```ts
import { test, expect } from '@playwright/test';
import { randomUUID } from 'crypto';
import { seedGame } from '../fixtures/seed';
import { openPlayerContext } from '../fixtures/players';
import { cleanupRun } from '../fixtures/cleanup';

test('notrump hand 5 deals with NT badge and betting UI', async ({ browser }) => {
  const runId = randomUUID();
  try {
    const game = await seedGame({ runId, scenario: 'notrump-hand-5' });
    const { page } = await openPlayerContext(browser, {
      session: game.hostSession,
      roomCode: game.roomCode,
      runId,
    });

    await expect(page.getByTestId('hand-counter')).toHaveText('Hand 5/20');
    await expect(page.getByTestId('trump-badge')).toHaveText(/NT|NO TRUMP/);
    await expect(page.getByTestId('bet-controls')).toBeVisible();
    await expect(page.getByTestId('player-tile')).toHaveCount(4);
  } finally {
    await cleanupRun(runId);
  }
});
```

The exact testIDs above are placeholders — the plan reads the current GameScreen to use the right ones, or adds them if missing. Adding testIDs to production components is in scope; refactoring component structure is not.

### Playwright config changes

`playwright.config.js` adds a second project entry:

```js
projects: [
  { name: 'e2e',      testDir: './tests/e2e' },
  { name: 'scenario', testDir: './tests/scenario' },
],
```

Per-test timeout, baseURL, viewport, etc. stay as today (scenario specs share the same per-test 30-min cap; the actual ~45s run finishes long before).

### npm scripts

Add to `package.json`:

```json
"test:scenario:local": "LOCAL_SUPABASE=1 HEADLESS=1 DEMO_URL=http://localhost:8082 playwright test --project=scenario"
```

No equivalent for the manual `:8081` flow yet — scenario specs assume the local-supabase stack because they need service-role access for cleanup.

### Cleanup invariant

Every scenario spec MUST:
- Generate its own `runId = randomUUID()` per test (not per suite).
- Pass that `runId` into every `seedGame` call.
- Run `cleanupRun(runId)` in a `finally` block (or `afterEach`).

The implementation plan codifies this in a comment at the top of `seed.ts`. Phase 5 will lint it via a custom check once the pattern is stable.

## Rollout

Single phase, single PR / commit chain on `main` (matches Phase 1 / Phase 2 cadence). Tasks in the implementation plan, expected count ~7-9:

1. Resolve the "how do bots get seated" question — read `createRoom`, `startGame`, edge function code.
2. Add testIDs to GameScreen if missing.
3. `tests/fixtures/supabaseAdmin.ts` + jest test for `assertLocalUrl` reuse.
4. `tests/fixtures/cleanup.ts` + integration test (insert via service-role, run cleanup, assert deletion).
5. `tests/fixtures/seed.ts` + integration test (call seedGame, assert hand 5 reached).
6. `tests/fixtures/players.ts` + integration test (open context, assert auth).
7. `tests/scenario/notrump-deal.spec.ts`.
8. `playwright.config.js` + `package.json` + `tests/README.md` updates.
9. End-to-end verification: `npm run test:scenario:local` passes; existing `npm run test:sp:local` still passes; `npx jest` still passes.

## Risks & mitigations

| Risk | Mitigation |
| --- | --- |
| No server-side path exists to seat bots — bot setup is purely UI-driven in `Quick Match` | Step 1 of plan investigates first. Fallback: add a small `addBotsToRoom` action to `game-action` (smallest possible production change, gated to service-role context). Documented as a fallback so it requires explicit decision, not silent expansion. |
| Anonymous auth session injection breaks if Supabase JS storage format changes | Plan reads the current expo client's auth-storage key explicitly rather than hard-coding it; if the key is computed at runtime, the helper computes it the same way. |
| FK cascades don't cover every dependent table | Plan inspects the schema before writing `cleanup.ts`; any non-cascading table gets an explicit `DELETE` in the helper. |
| Bots time out on hands 1-4 and the seed hangs | seedGame has a hard 90s deadline. If `hand_number === 5` not reached, throw a clear error with the last snapshot for debugging. |
| Test pollutes the dev `:8081` DB if someone runs the spec without `LOCAL_SUPABASE=1` | `supabaseAdmin()` calls `assertLocalUrl` — same prod-leak guard `globalSetup` already uses. Refuses to run against any non-local URL. |

## Open questions resolved during brainstorming

- **Scope** → strict POC: one scenario, one spec, one minimal `players` helper.
- **Seed strategy** → drive edge functions; no direct INSERTs.
- **Cleanup** → prefix existing columns (`display_name = test_<runId>_*`); no schema change.
- **Speed vs purity** → accept ~30-60s seed runtime; revise the spec's original "~5s" expectation.
